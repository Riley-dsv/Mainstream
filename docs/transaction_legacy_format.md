# Legacy Transaction File Format

## Purpose

`transaction_legacy.dat` is the fixed-width sequential file produced by the COBOL batch.
Each line represents one raw transaction record.
The file is designed to simulate a legacy batch input format before transformation into modern Kafka JSON events.

## Design Principles

- One transaction per line.
- Fixed-width positional format.
- Numeric fields are zero-padded on the left.
- Alphanumeric fields are space-padded on the right.
- Dates use `YYYYMMDD`.
- Times use `HHMMSS`.
- Monetary amounts are stored as integers in the smallest currency unit.
- The file only contains syntactically valid raw transactions.

## Record Layout

Each record has a total length of **162 characters**.
Spaces between logical blocks are part of the format and must be preserved.

| Field | Start | End | Length | Type | Format / Rule |
|:---:|:---:|:---:|:---:|:---:|:---:|
| transaction id | 1 | 31 | 31 | alphanumeric | unique transaction identifier, space-padded right if needed |
| filler-1 | 32 | 32 | 1 | literal | space |
| operation date | 33 | 40 | 8 | numeric | `YYYYMMDD` |
| filler-2 | 41 | 41 | 1 | literal | space |
| operation time | 42 | 47 | 6 | numeric | `HHMMSS` |
| filler-3 | 48 | 48 | 1 | literal | space |
| transaction type | 49 | 51 | 3 | alphanumeric | example: `TRF` |
| filler-4 | 52 | 52 | 1 | literal | space |
| amount | 53 | 64 | 12 | numeric | integer amount in smallest currency unit, zero-padded left |
| filler-5 | 65 | 65 | 1 | literal | space |
| currency | 66 | 68 | 3 | alphabetic | ISO 4217, example: `EUR` |
| filler-6 | 69 | 69 | 1 | literal | space |
| sender iban | 70 | 96 | 27 | alphanumeric | sender account identifier |
| filler-7 | 97 | 97 | 1 | literal | space |
| receiver iban | 98 | 124 | 27 | alphanumeric | receiver account identifier |
| filler-8 | 125 | 125 | 1 | literal | space |
| reference | 126 | 145 | 20 | alphanumeric | business label, space-padded right |
| filler-9 | 146 | 146 | 1 | literal | space |
| source system | 147 | 154 | 8 | alphanumeric | example: `BATCH001` |
| filler-10 | 155 | 162 | 8 | literal | trailing spaces |

## Field Definitions

### transaction-id
Unique identifier generated by the legacy batch.

Expected pattern:

`TXN-YYYYMMDDHHMMSS-<workerId>-<sequence>`

Example:

`TXN-20260422162600-098-000000178`

Rules:
- `TXN` prefix is constant.
- Timestamp uses `YYYYMMDDHHMMSS`.
- `workerId` is a 3-digit identifier.
- `sequence` is a 9-digit counter.
- Field is left-aligned and padded with spaces to length 31 if needed.

### operation-date
Business operation date in `YYYYMMDD` format.

### operation-time
Business operation time in `HHMMSS` format.

### transaction-type
Transaction business code.

Allowed initial values:
- `TRF` = transfer

### amount
Amount expressed as an integer in the smallest currency unit.

Examples:
- `000000120000` = 1200.00 EUR
- `000000085500` = 855.00 EUR
- `000000009999` = 99.99 EUR

Rules:
- exactly 12 digits
- no decimal separator
- positive values only in the raw legacy file for now

### currency
3-letter ISO 4217 currency code.

Initial allowed value:
- `EUR`

### sender-iban
Text account identifier of the sender.
Stored as plain text in the legacy record.

### receiver-iban
Text account identifier of the receiver.
Stored as plain text in the legacy record.

### reference
Short business label describing the payment.

Examples:
- `paiement loyer`
- `facture energie`
- `abonnement internet`

Rules:
- max length: 20 characters
- left-aligned
- right-padded with spaces
- avoid accents and special characters

### source-system
Identifier of the producer system.

Initial fixed value:
- `BATCH001`

## Encoding and File Rules

- File format: plain text
- Encoding: UTF-8 preferred, restricted to simple ASCII-compatible characters in practice
- One record per line
- Preserve trailing spaces if your tooling allows it
- Do not insert tabs
- Do not use separators other than the fixed positional spaces defined in the layout
